1 · Overview — Feature, Audience & Dependencies

Feature Summary

mohan-dotfiles is a personal machine setup distributed as a reproducible Nix flake. It declaratively defines and deploys every configuration file, package, and tool across Linux and WSL environments, ensuring every fresh machine clone installs identically pinned versions with a single command. The system treats live files under $HOME as read-only Nix store symlinks, enforces configuration drift detection, and preserves machine-local secrets outside version control.

Audience & Actors

Functional Requirements

ID Requirement Acceptance Criteria
FR1 Declarative configuration in Nix All config files (.zshrc, .gitconfig, claude/CLAUDE.md, etc.) are generated by Home Manager from nix/*.nix; edit Nix, not the live files
FR2 Reproducible deployment flake.lock pins nixpkgs and home-manager versions; running setup.sh on machine A and machine B yields identical $HOME with same packages and versions
FR3 Drift detection and recovery setup.sh doctor scans every managed file and detects edits outside Nix; setup.sh with no args reverts drifted files to Nix-managed symlinks, saving hand-written copies as *.hm-backup
FR4 Machine-local secrets isolation Secrets never enter git; ~/.zshrc.local (untracked) is sourced last by managed zshrc; setup-packages.sh writes to ~/.config/mohan-dotfiles/packages-config.nix
FR5 Modular, scoped config nix/home.nix imports one module per concern: packages.nix, zsh.nix, git.nix, claude.nix, copilot.nix, nvim.nix, pi.nix; changes to one module do not require editing others
FR6 CLI tool discovery and updates setup.sh installs Nix if missing; setup-packages.sh presents a checklist (Docker, Python, Copilot, etc.); re-running setup-packages.sh and setup.sh re-installs changed selections
FR7 Hook regression testing claude/hooks/test-hook.sh selftest and copilot/hooks/test-hook.sh selftest run in sandbox; flake checks enforce that both pass before CI success
FR8 Cross-machine account flexibility flake.nix reads $USER at eval time with --impure; same checkout works under any username without edits

Non-Functional Requirements

Attribute Measurable Target
Reproducibility 100% identical outputs across machines given same flake.lock, evaluated with --impure and same $USER
Deployment speed setup.sh completes in under 5 minutes on a modern machine (after Nix install on first run)
Drift detection latency setup.sh doctor completes scan of all managed files in under 2 seconds
CI/CD check duration nix flake check --impure (home eval + hook tests + setup.sh lint/doctor) completes in under 60 seconds on GitHub Actions runner
Availability setup.sh has no external dependency other than GitHub (for git clone); Nix installer fallback to Determinate Systems
Observability Every setup.sh command logs its action; doctor reports specific file paths with drift type; setup-packages.sh shows selected vs. unselected in UI

Out of Scope

  • NixOS system configuration (this flake is Home Manager only, runs on non-NixOS systems)
  • Multi-user machine setup (designed for single user account per machine)
  • Automatic deployment via CI/CD (manual setup.sh invocation required; no push-to-deploy)
  • Custom binary provisioning (only packages in nixpkgs or pre-built binaries via Home Manager activation scripts)
  • Machine-to-machine sync of generated files (setup-packages.sh selections are local; ~/.zshrc.local secrets never synced)

Key Relationships

  • flake.nix → nix/home.nix: Home Manager module system; nix/home.nix imports all concern-specific modules (packages.nix, zsh.nix, etc.)
  • nix/* → config files: Each Nix module generates Home Manager configuration that writes files from claude/, copilot/, nvim/, pi/, zsh/ into $HOME
  • setup.sh → Home Manager activation: Reads current flake.lock and runs home-manager switch --flake . --impure to apply all changes atomically
  • setup.sh doctor → state/nix/profiles/home-manager/home-files: Compares every link point in $HOME against the current generation to detect drift
  • setup-packages.sh → ~/.config/mohan-dotfiles/packages-config.nix: Writes user selections (yes/no per package) into configPath; next setup.sh applies it
  • CI/CD checks → flake.nix: nix flake check --impure runs home eval check, hook regression tests, and setup.sh lint/doctor in sandbox

State Version & Schema

State Version: 25.05 (set once in nix/home.nix, do not change after first activation; Home Manager uses this to apply state-schema migrations)

Configuration Sources:

  • Version control (flake.nix, nix/*.nix, claude/, copilot/, nvim/, pi/, zsh/) — committed, reproducible, reviewed
  • Machine-local (flake.lock — generated, committed after `nix flake update`; ~/.config/mohan-dotfiles/packages-config.nix — generated by setup-packages.sh, not committed)
  • Secrets outside version control (~/.zshrc.local — sourced last, never committed)

Schema Changes: None expected for this flake; if nixpkgs updates Home Manager and a migration is needed, nix/home.nix's stateVersion will guide it. Schema is forward-locked by flake.lock pinning.

CLI Entry Point: setup.sh

Command Args Input Output Exit Codes
setup.sh (no args) Current flake.nix, flake.lock, nix/*.nix Nix installed (if missing); home config activated; login shell set to zsh; drift audit run; *.hm-backup files for any edits found 0 if success; 1 if Nix install fails or drift audit fails
setup.sh doctor doctor Current generation symlinks in $HOME Line-by-line report of every managed file (✓ OK, ✗ MISSING/DRIFTED); links checked from ~/.local/state/nix/profiles/home-manager/home-files 0 if all files match expected state; 1 if any drift detected
setup.sh upgrade upgrade flake.nix Runs nix flake update to bump flake.lock inputs; then runs setup.sh; updates flake.lock in git (user must commit) 0 if update + switch success; 1 if update or switch fails

CLI Entry Point: setup-packages.sh

Command Input Output Notes
setup-packages.sh Current ~/.config/mohan-dotfiles/packages-config.nix (if exists; else defaults to all false) Interactive prompts (yes/no per package); writes selections to ~/.config/mohan-dotfiles/packages-config.nix; user must then run setup.sh to apply One y/n prompt per optional package (Docker, Python, Copilot CLI, etc.); selections persist across re-runs

Nix CLI Integration (User-Facing)

Note: Users do not call nix commands directly; setup.sh wraps them. For reference:

# What setup.sh runs internally:
nix flake check --impure                  # validates home eval, hook tests, setup.sh lint/doctor
home-manager switch -b hm-backup --flake . --impure  # activates config, backs up drifted files
nix flake update                          # bumps flake.lock (in setup.sh upgrade)

The --impure flag is mandatory because flake.nix reads $USER from the environment at eval time, making outputs account-specific.

Shell Prompt & Terminal UX

Prompt Design: zsh/prompt.zsh is a custom from-scratch prompt styled after old CRT monochrome monitors. Every fresh interactive terminal shows a one-line typewriter-animated greeting. The prompt itself:

  • Shows full working directory (not shortened)
  • Displays git branch with emoji status: 🌿 clean, 🔥 dirty, ↑N/↓N for commits ahead/behind
  • Git status computed in background job (zle -F) so slow repos never block typing; shows spinner (⠋⠙⠹) while pending
  • RPROMPT shows ✅/❌ based on last exit code; ⏱ Ns if command ran 3+ seconds
  • Clock in terminal window title (updated every prompt) instead of repeating per line
  • Each field gets its own step on green phosphor ladder for scannability

Edit zsh/prompt.zsh directly (plain zsh, no Nix escaping) and re-run setup.sh to apply changes.

User manually edits ~/.zshrc setup.sh doctor detects: readlink -f $HOME/.zshrc does not match expected store path ~/.zshrc is no longer a symlink; it is a hand-written copy. doctor reports DRIFTED Run setup.sh (with no args) to re-link; hand-edited copy is saved as ~/.zshrc.hm-backup

Flake Check Pipeline (CI/CD)

Every push to GitHub triggers .github/workflows/ci.yml, which runs four checks. All must pass before merge:

Check Command Purpose Failure Outcome
home-eval nix flake check --impure (checks.home) Instantiate (not build) every derivation in the home config tree; validates all Nix syntax, Home Manager options, and file references without downloading gigabytes of GUI libraries PR blocked; error message shows the offending module and line
claude-hooks-selftest nix flake check --impure (checks.hooks-selftest) Run claude/hooks/test-hook.sh selftest in a sandbox HOME with JSON payloads on stdin, verifying all hook guards and transformations work correctly PR blocked; hook behavior is untrusted
copilot-hooks-selftest nix flake check --impure (checks.copilot-hooks-selftest) Run copilot/hooks/test-hook.sh selftest (shell port) in sandbox, verifying Copilot CLI hook equivalents work PR blocked; Copilot hook port is out of sync
setup-script nix flake check --impure (checks.setup-script) Lint setup.sh (bash -n, shellcheck), then run setup.sh doctor against a synthetic Home Manager profile (clean, hand-edited, deleted) to verify drift detection works PR blocked; setup.sh has syntax errors or doctor is broken

Nix Modules & Derivations

Key Modules (nix/*):

  • packages.nix: home.packages list (all CLI tools, utilities); activation scripts for npm global CLI and native Nix post-activation hooks
  • zsh.nix: home.sessionVariables, home.file entries for managed zsh config, oh-my-zsh config, symlink to zsh/prompt.zsh
  • git.nix: programs.git options (user name/email, aliases, core settings)
  • claude.nix: home.file entries for ~/.claude/ tree (CLAUDE.md, hooks, skills, commands, agents)
  • copilot.nix: home.file entries for ~/.copilot/ tree (copilot-instructions.md, hooks, agents, skills) — shell port of claude/
  • nvim.nix: home.file with symlink from ~/.config/nvim directly to ~/REPO/mohan-dotfiles/nvim (LazyVim checkout)
  • pi.nix: home.file entries for ~/.pi/agent/extensions/ (TypeScript hooks + pipeline-panel dashboard)

Output Locations: All derivations are built to /nix/store; Home Manager reads from store and creates symlinks in ~/.local/state/nix/profiles/home-manager/home-files/, then links those into $HOME/.* paths.

Versioning & State Lock (flake.lock)

flake.lock: Records exact git revisions for nixpkgs (release-25.05) and home-manager (release-25.05). Committed to the repo. Every setup.sh run reads flake.lock and fetches exactly those versions, ensuring reproducibility across machines and time.

Bumping versions: Run setup.sh upgrade, which executes nix flake update (updates flake.lock), then setup.sh (applies new versions). User must git add flake.lock && git commit to finalize.

stateVersion: Home Manager migration tracking (set in nix/home.nix to 25.05). Do not change after first activation; Home Manager uses this to apply state schema migrations on future minor versions.

Post-Activation Hooks & Scripts

Home Manager supports activation scripts that run after all files are symlinked. nix/packages.nix defines:

  • npm global CLI setup: Installs packages to $HOME/.npm-global (Claude Code, pipeline-worker CLI) if missing; native activation script runs post-activation
  • Claude Code native setup: Installs Claude Code via official installer (not Nix) if missing; runs only once

These run idempotently; they check if the binary already exists before downloading.

Alert Scenarios

setup.sh doctor reports drift: One or more managed files have been edited outside Nix or deleted. Recovery: run setup.sh (with no args) to re-link; hand-edits are saved as *.hm-backup. Review the backup, merge any necessary changes back into the .nix source, commit, and re-run setup.sh.

nix flake check fails: One or more CI checks failed (usually a Nix syntax error or hook test regression). Recovery: fix the offending .nix file or test case, commit, and push; CI re-runs automatically.

home-manager switch fails: Activation package failed to build or activate (e.g., missing dependency, permission error). Recovery: run setup.sh doctor to see current state; if broken, check for partial symlinks or /nix/store path issues; retry setup.sh once root cause is fixed.

Nix install fails: Network or installer issue. Recovery: retry setup.sh once network is restored; ensure_nix is idempotent and skips if nix is already in PATH.

Incident Flow (Drift Detected → Resolution)

Manual observability (no automated dashboard): Users run setup.sh doctor on demand to audit current state. Output is human-readable and actionable. Architectural Decisions

Decision Options Considered Chosen & Rationale Date
ADR1: Home Manager vs. NixOS (A) Full NixOS system flake; (B) Home Manager only on non-NixOS systems; (C) Hybrid (NixOS + Home Manager) Chosen: (B) Home Manager only. Rationale: Mohan uses Linux/WSL (non-NixOS) workstations. NixOS requires system-level access and is overkill for user-level config. Home Manager is portable, lightweight, and sufficient. NixOS machines would need a separate flake. Project inception
ADR2: Symlink-based immutability (A) Copy files, allow edits, sync back on setup.sh; (B) Symlinks (read-only store), detect/reject drift; (C) Watch files + auto-sync Chosen: (B) Read-only symlinks + drift detection. Rationale: Makes the contract explicit: config lives in nix/ and git, live files are read-only. Drift detection (doctor) alerts without silently clobbering user edits. *.hm-backup preserves hand-edits for review. Simpler than watches and more intentional than auto-sync. Project inception
ADR3: setup.sh as single CLI entry point (A) Multiple scripts (setup.sh, apply.sh, doctor.sh, upgrade.sh); (B) One setup.sh with subcommands; (C) Nix flake commands (nix run .#setup, etc.) Chosen: (B) setup.sh with subcommands (doctor, upgrade). Rationale: Familiar bash pattern, works without Nix in PATH initially, no need to understand flake apps. shell history shows setup.sh doctor (clear intent) better than nix run .#doctor (indirection). setup.sh is the contract between user and system. Early development
ADR4: Hook porting across tools (A) One hook implementation per tool (Claude, Copilot, Pi); (B) Single implementation, adapt at call site; (C) Shared library (Nix module or shell lib) Chosen: (A) Three ports (shell bash for Claude/Copilot, TypeScript for Pi). Rationale: Each tool has its own hook/extension API (JSON payloads differ, return formats differ). Porting gives us tool-specific ergonomics and error handling. Keeps ports in sync via documented rules in CLAUDE.md. Single implementation would be more complex abstraction than three targeted versions. Feature expansion
ADR5: --impure flake eval (A) Pure flake (--pure), read username from default file (complex); (B) --impure with $USER at eval time (current); (C) Parameterized username via flake args Chosen: (B) --impure with builtins.getEnv "USER". Rationale: Simplest pragmatic solution. Flake is not "pure" in the academic sense (depends on env), but it is deterministic given the same $USER and flake.lock. Avoids verbose configuration per machine. Trade-off documented in flake.nix comments and README (--impure is required). Early development

Risks & Mitigations

Risk Likelihood Impact Mitigation
R1: flake.lock drift (user runs nix flake update without committing, then clones on another machine with old flake.lock) Low High (machines get different packages; reproducibility lost) Enforce: setup.sh upgrade requires explicit git commit. CI checks that flake.lock is committed. CLAUDE.md documents the contract. User discipline.
R2: Secrets in repo (user accidentally commits ~/.zshrc.local or GitHub token to git) Low Critical (credentials exposed) ~/.zshrc.local is in .gitignore. Setup.sh migrate_pre_nix_dotfiles warns about hand-written files. CLAUDE.md explicitly lists that secrets go in ~/.zshrc.local, never in Nix. User discipline + git pre-commit hooks (not yet implemented).
R3: Home Manager breaking change (home-manager release-25.05 introduces schema migration that breaks stateVersion 25.05) Very Low High (activation fails; user config broken) stateVersion is pinned and intentionally immutable. Home Manager project maintains backward compat within a release series. If upgrade needed, user bumps stateVersion + runs setup.sh; schema migration runs automatically.
R4: Nix install failure (Determinate Systems installer is unavailable or changed) Low High (new machine setup blocked) Fallback: document alternative Nix installers (NixOS.org official, Nixpkgs multi-platform). setup.sh ensure_nix checks PATH first, so pre-installed Nix is used if available. Manual install is always possible.
R5: Large /nix/store growth (multiple generations accumulate; disk fills) Medium Medium (disk space exhausted, setup.sh fails) Home Manager stores old generations in ~/.local/state/nix/profiles/. User can manually run nix-collect-garbage to clean old generations. Document in README (not yet implemented).
R6: Drift detection false positives (setup.sh doctor reports drifted file when symlink is actually correct) Very Low Low (confusing output, but re-running setup.sh resolves it) doctor compares readlink -f output for exactness. Symlinks are created by Home Manager, so link targets are deterministic and testable. Hook tests validate doctor logic; CI runs doctor tests on every commit.

Future Enhancements

  • Automated garbage collection: setup.sh could run nix-collect-garbage --delete-older-than 30d to keep /nix/store under a size threshold
  • Distributed machine state: Sync ~/.zshrc.local (or parts of it) across machines via encrypted Git repo or secure key-value store
  • Pre-commit hooks: Add .git/hooks/pre-commit to scan for committed secrets (GitHub token, AWS keys) before allowing push
  • Automated drift repair: setup.sh doctor with --auto-repair flag to fix drifts without user interaction (currently user runs setup.sh)
  • Machine onboarding wizard: Interactive TUI that guides setup-packages.sh and setup.sh in sequence
  • Declarative optional package groups: Define "base" (minimal), "dev" (Docker, Python), "enterprise" (Copilot, LocalScribe) groups; user selects group instead of individual y/n
